为什么需要写博客和写作

在这里说一下我为什么突然开始觉得我需要写博客，最开始我是不想写博客的，有好几点原因：1,懒，嗯，这是首要因素。2,我觉得写博客需要花费大量的时间和精力，然而我想把这些时间用来留着看其他书和学习（我在心里是这么跟自己说的，然而多出来的时间我既没看书也没有学习）。3,我觉得自己写得不好，自己想写的东西很多都被别人写过了，而我想写些不一样的，更牛逼的东西（嗯，这个想法是好的…但是记录不一定是都给别人看的，也给自己看，这样可以自己评价自己）

直到最近在看一本书，叫做《暗时间》，作者是刘未鹏，里面建议说每个人都应该自己学着去写博客，这样会让你不断地学会如何思考，因为在给别人介绍知识点的时候，同时自己也就需要将脑海的知识点梳理一遍（像费曼方法），这样不仅加深了记忆，也能使自己对知识点的理解上一个台阶。

其中在《暗时间》中有一句话让我特别深刻的是：我们每个人的生命就像一个个沙漏，每个人的沙漏里装的沙子总量是相当的，不同的是，有的人的沙漏颈部较细，有些人的沙漏的颈部较粗。那些颈部较细的沙漏能够抓住每一粒时间之沙，虽然沙子总量一样，但却相对拥有更长的生命。

这句话的大概意思就是每个人的生命的长度也许大致是差不多的（平均在70-80岁左右），但是有些人的生命却拥有更广的宽度和深度。如果能利用 自己平时吃饭，走路，洗手，旅游这些暗时间来思考并总结自己学过的知识，然后把它们以写作的形式记录下来，自己不就相当于比别人多了很大一截的时间了吗？

所以，我觉得开始写博客了：1,看书很重要，但不要只是计算自己看了多书，自己看了那么多书，却不见得自己的领悟有多深，表面上花的时间不少，收益却见不得有多大。因为只是单纯地接收知识却没有经过自己的思考，质疑，反刍，实践的话，很多东西是没有办法融入到你的身体里面的，这也许就是韩寒说的那句话，听过很多道理，却没有过好这一生。因为道理谁都懂，却不是人人都去思考为什么需要这样做，为什么自己做不到。上下班不打招呼的，因为这样反而打扰人家工作
2,我觉得写博客除了能加强自己的思考，还有一个好处就是作为自己的笔记本，自己可以回过头来审视自己的思考，通过自己的不断修改，找出自己的思维误区，完善自己。
3,作为自己提问以及抒发情感的地方，文字是一种很奇妙的东西，它可以让你有各种各样的表达，并记录下来。同时我觉得提问要比解答重要多了，提问在某种意义上也是一种质疑，纵观历史，社会的进步就是靠提问和质疑促进的，没有牛顿对苹果为什么会落地的提问，也就没有后来万有引力的发现，很多看似合理的事情，其实有着很多不合理的情况，但是如果你根据这些你觉得“不合理”的情况深究下去，相信你能最终找到合理的解释的。所以，我要做那个好奇的，喜欢提问的，“捡贝壳”的小孩。

在CDH测试集群升级spark2.X

在CDH5.14.0集群中，默认安装的spark是1.6版本，这里需要将其升级为spark2.x版本。经查阅官方文档，发现spark1.6和2.x是可以并行安装的，也就是说可以不用删除默认的1.6版本，可以直接安装2.x版本，它们各自用的端口也是不一样的。第一次实验的时候安装2.1.0.cloudera2版本，但是在cloudera manager的web页面显示 无法安装，觉得可能是跟CDH版本冲突的问题（虽然从官网上看来版本是符合的…）,后修改spark版本为：2.2.0.cloudera2，安装成功！运行测试例子：./bin/run-example SparkPi 10 > ～/Sparkpilog.txt亦成功。下面介绍安装过程：

spark1.6服务可以与spark2共存于同一个集群上。两个服务配置在同一个yarn集群上运行不会发生冲突。
官方升级文档

环境准备：

1,与CDH版本对应
2,与Cloudera Mnanager 版本对应
3,scala版本应为2.11
4,需要安装有python环境
5,需要配置jdk8
参考链接

集群CDH版本为5.14.0，故选择sprk版本为：2.2.0.cloudera2。

以下对每台机器都做相同的操作：
下载csd包到/opt/cloudera/csd目录，并且修改文件的用户和组。注意如果本目录下有其他的jar包，把删掉或者移到其他目录：
cd /opt/cloudera/csd

sudo wget http://archive.cloudera.com/spark2/csd/SPARK2_ON_YARN-2.2.0.cloudera2.jar

sudo chgrp cloudera-scm SPARK2_ON_YARN-2.2.0.cloudera2.jar

sudo chown cloudera-scm SPARK2_ON_YARN-2.2.0.cloudera2.jar

下载parcel包到机器的/opt/cloudera/parcel-repo目录下。注意，如果有其他的安装包，不用删除 ，但是如果本目录下有其他的重名文件比如manifest.json文件，把它重命名备份掉。然后把那3个parcel包的文件放在这里。

CDH集群升级spark需要两个安装文件：
一个叫做csd文件(在CDH集群的/opt/cloudera文件下可见)，另一个是spark2.x的parcel安装包

cd /opt/cloudera/parcel-repo

sudo wget http://archive.cloudera.com/spark2/parcels/2.2.0.cloudera2/SPARK2-2.2.0.cloudera2-1.cdh5.12.0.p0.232957-trusty.parcel

sudo wget http://archive.cloudera.com/spark2/parcels/2.2.0.cloudera2/SPARK2-2.2.0.cloudera2-1.cdh5.12.0.p0.232957-trusty.parcel.sha1

sudo wget http://archive.cloudera.com/spark2/parcels/2.2.0.cloudera2/manifest.json

sudo mv SPARK2-2.2.0.cloudera2-1.cdh5.12.0.p0.232957-trusty.parcel.sha1 SPARK2-2.2.0.cloudera2-1.cdh5.12.0.p0.232957-trusty.parcel.sha

sudo chgrp cloudera-scm ./*

sudo chown cloudera-scm ./*

在cloudera manager运行：

$ sudo service cloudera-scm-server restart

奇技淫巧

本文主要是记录一些本人在日常开发中遇到的一些小问题和相应的解决方法,希望有遇到类似问题的网友能得到快速的解决方案。

1,在用ssh远程连接服务器的时候，如果有一段时间没有操作界面，会经常遇到终端界面卡住的情况，这是ctrl+d都无济于事，只能关闭窗口再重新连接，后来我在网上看到有网友键盘输入～.就中断了连接，解决了。瞬间变有种很优雅的感觉。按照网友在man ssh的时候，也查到了关于～.命令的解释，就是disconnect，断开连接的意思。

2,今天在远程登录自己的ETL服务器的时候，突然发现在自己的服务器的文件颜色高亮没有了，在网上查后知道，可以根据dircolors的功能来修改配置，在用户目录的.bashrc文件中加入以下内容：
$ vim ~/.bashrc

enable color support of ls and also add handy aliases

if [ -x /usr/bin/dircolors ]; then

eval “dircolors -b“

alias ls=’ls –color=auto’

alias dir=’dir –color=auto’

alias vdir=’vdir –color=auto’

alias grep=’grep –color=auto’

alias fgrep=’fgrep –color=auto’

alias egrep=’egrep –color=auto’

fi

#—————- < my_dircolors Alyan @2009.03 > ——#

# .c .h .cc .cpp 文件 加粗显示

# README README.txt 加亮红色

# Makefile 加下划线

export LS_COLORS=”$LS_COLORS:README=01;31:README.txt=01;31:*Makefile=

04;38:.h=01;26:.c=01;29:.c c=01;26:.cpp=01;29:”

由于linux内核过多引起的根盘inode溢出的问题，下面介绍一下我排除问题的思路：
首先,需要了解的是，inode是专门用来记录linux上的文件元信息和文件索引的，刚开始收到zabbix的监控报警，在hadoop集群机器上执行df -h命令，发现根盘并没有满，但是执行df -i 命令，发现inode数量占用超过90%，起初以为是跟hadoop有关，但是hadoop集群数据盘占用的inode是很少的（每个数据块(block)128M的原因）。故排除跟hadoop的关系，后执行：
for i in `ls -1A | grep -v \”\.\./\” | grep -v \”\./\”`; do echo \”`find $i | sort -u | wc -l` $i\”; done | sort -rn | head -10
找到根目录下文件数量排在前10位的文件夹：
892618 proc
383545 usr
311594 home
42700 sys
15317 lib
10466 var
1970 etc
498 run
365 dev
346 boot

proc为系统硬件文件目录，因为不够了解所以没敢动，后来进入/usr/libx86_64-linux-gnu目录执行：ls -lR|grep “^-“| wc -l 发现，在这个目录下的linux内核文件占用了很多文件inode，所以判断是因为文件内核过多的问题，故：
先执行uname -a
查看本机多用的linux内核（生产集群，必须要小心看清楚，，）
然后对应/usr/libx86_64-linux-gnu目录，找出不用的linux内核，执行：
sudo apt-get remove $linux-core-version
讲多余的linux内核删除，执行df -i
inode占用为35%，问题解决

Presto安装过程中遇到的一些问题

Presto是一个开源的分布式查询引擎，适用于交互式分析查询。

我司原来使用的HDFS交互查询工具是Hive，使用的计算引擎是MapReduce，但是由于MapReduce的计算过程相对我们的需求来说有些慢，所以我们决定换上号称在同等的其他条件下可以比Hive快10倍的Presto。

在Presto的安装过程中，根据安装链接 ，我们跟着步骤走，到启动presto的时候，报了无法连接hive的错误，经查找错误，发现是配置的问题，presto是通过thrift(通信协议)来连接Hive的，默认的端口为9083,所以需要把presto的/etc/catalog下的配置文件hive.properties中加上如下配置：
connector.name=hive-hadoop2
hive.metastore.uri=thrift://shymaster1:9083
hive.config.resources=～/hadoop-2.6.4/etc/hadoop/core-site.xml,～/hadoop-2.6.4/etc/hadoop/hdfs-site.xml

同时，保证在Hive的配置hive-site.xml中开启了thrift服务，端口亦为9083。

用命令：
./presto –server localhost:(presto通信端口) –catalog hive –schema default
（其中，presto命令脚本本质是一个启动presto的jar包 ）启动presto，运行命令，show tables;没有报错，即已安装完成。

Hive无法远程连接metastore的问题

##hive客户端在远程连接metastore所在数据库时报错

报错起因：本人所维护的一台AWS的服务器(即metastore所在服务器)被回收了(用作测试机，有被回收的可能，故价格低一些)。所以在重新配置好mysql的metastore、mysql中hive的用户权限后，本从用hive远程客户端连接metastore时报以下错误：
Exception: Unable to instantiate org.apache.hadoop.hive.ql.metadata.SessionHiveMetaStoreClient

分析：由错误初步判断是网络不通引起的，所以从网络连接这块的检查下手，首先在hive客户端机器master2(在我这里hive客户端为master2，hive的metastore所在机器为master1)中执行命令：telnet master1 3306，结果不同，不返回“Escape character is ‘^]’”。

故到metastore所在的master1机器上查看3306端口是否已开放，
执行命令：netstat -ntpl |grep 3306
返回结果是：tcp 0 0 127.0.0.1:3306 0.0.0.0:* LISTEN
3306端口是打开的，但是发现3306的端口绑定的ip地址是127.0.0.1的，也就是说在非本机的telnet 3306是没办法有返回消息的。
故从mysql的IP的绑定google，发现只要在mysql的配置文件 (我的是在 /etc/mysql/)下修改配置文件.cnf后缀的配置文件中将bind-address = 127.0.0.1 的那一行注释掉就可以了。

注意：在mysql的配置文件夹下有好几个.cnf的配置文件，里面均有可能配置有bind-address，需要全部都注释掉！本人就是在这里疏忽大意，漏改了配置文件！导致花了几个小时。

总结：我在这次错误中断断续续踩了一个早上的坑。回过头来看，主要是有由于自己对系统的了解不够深入和对mysql配置的不熟悉导致的，写此文警醒自己夯实基础。

根据hexo && github 搭建自己的博客

##Github Pages
Github Pages 原本是用于托管在github上的项目的，由于它提供免费稳定的空间，所以可以用来搭建博客。

每个github帐号只能有一个仓库来存放自己的个人主页，而且仓库的名字必须是username/username.github.io,这个特殊的命名约定，让别人可以通过http://username.github.io来访问个人主页。

需要注意的是，个人网页的网站内容必须是在master分支下的。

##部署Hexo到Github Pages

意思就是把Hexo本地生成的博客部署到github上，然后可以通过http://username.github.io来访问博客的内容

首先，需要在自己的本地建立一个与github相连接的项目，然后创建一个archive文件夹。
$ git clone git@github.com:username/username.github.io.git
/username.github.io$ mkdir archive

然后下载安装Hexo(需要提前安装好node.js等插件)，并根据Hexo生成博客模板
/username.github.io$hexo archive

在archive文件中的 _config.xml末尾添加：
deploy:
type: git
repo: git@github.com:username/username.github.io.git
branch: master

然后执行：
/username.github.io$ hexo d
完成部署。

最后，将自己的本地项目推送到github上，然后打开http://username.github.io即可。
